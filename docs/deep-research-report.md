## North star and design principles

The project should be designed around a stable product core and changing experiences. In practice, that means the public-facing brand can stay AI-oriented for now if that helps users understand the offer, while the internal platform, repo structure, package naming, and service contracts become topic-neutral immediately. This reduces future migration pain and avoids embedding ŌĆ£AIŌĆØ into every namespace, database table, or API path. Twelve-Factor principles support that separation by encouraging explicit dependencies, externalized config, and portability across environments, while OpenFeature and MCP both support standardized interfaces that survive tooling or vendor changes over time. The project will last longer if you define a small set of durable platform abstractions now. The most important ones are: **Topic**, **Content unit**, **Skill**, **Assessment**, **Learner profile**, **Interaction event**, **Recommendation**, and **Policy decision**. If your model today is something like `AICourse`, `AIPrompt`, or `AISkillPath`, you should treat those as topic-specific implementations of broader concepts. A durable north star also needs measurable system goals. For this project, I would define success at three layers. ## Reference architecture for multi-site and multi-pipeline growth

The cleanest long-term shape is a **modular monorepo** at first, with clear package boundaries, and the option to split later only when scale or team structure demands it. Early on, a monorepo makes shared schemas, shared CI policies, shared evaluations, and shared release gates much easier to govern. As service count grows, a software catalog such as Backstage becomes useful because it centralizes component ownership, APIs, documentation, and dependencies across many services without forcing every team into one runtime or one deployment unit. ## Personalization beyond the chatbot layer

The most durable personalization strategy is to start with **behavioral intelligence**, not with highly sensitive profile inference. In other words: personalize from observed learning signals, not from invasive demographic assumptions. Educational research supports this direction. Bayesian Knowledge Tracing models learner state probabilistically over time, and Deep Knowledge Tracing showed that neural sequence models can materially improve prediction of student performance and support intelligent curriculum design. In parallel, Retrieval-Augmented Generation remains the right pattern for grounding generation on trusted, updateable knowledge rather than relying purely on model memory. ŅłĆciteŅłéturn9search8Ņłéturn9search1Ņłéturn9search0Ņłéturn9search2Ņłéturn9search13Ņłü

This leads to a layered personalization roadmap. At the **ranking layer**, personalize search results, suggested lessons, and explanation examples. At the **curriculum layer**, recommend the next best content unit, practice cycle, or assessment based on learner state. At the **generation layer**, adapt tone, depth, hints, and scaffolding style. At the **retention layer**, personalize review timing, reminders, and streak recovery. DuolingoŌĆÖs Birdbrain is a strong proof point that learner modeling can sit underneath lesson selection rather than only inside a chat experience. Khanmigo, Coursera Coach, and Quizlet Q-Chat show parallel patterns in tutoring, interactive instruction, and adaptive prompting. ŅłĆciteŅłéturn19search3Ņłéturn19search7Ņłéturn19search11Ņłéturn18search1Ņłéturn18search10Ņłéturn18search5Ņłéturn8search1Ņłéturn19search8Ņłü

Operationally, I would build personalization in three layers. First, a **profile store** that holds stable preferences and learning-state summaries. Second, an **event stream** that records granular interactions such as impressions, clicks, completions, failures, retries, dwell time, and explicit feedback. Third, a **feature store** that materializes training and inference features consistently for rankers or recommendation models. Feast is purpose-built for this kind of consistency, and MLflow gives you the experiment tracking and evaluation machinery you need to compare algorithms and prompts over time. A good ŌĆ£phase oneŌĆØ personalization stack would stay simple: heuristic ranking plus a few interpretable features such as recent topic engagement, completion depth, correctness streak, and declared goals. This sequence matters because premature ML often increases system complexity before you have reliable instrumentation, evaluation sets, and rollback discipline. OpenAIŌĆÖs eval guidance and MLflowŌĆÖs agent and LLM evaluation guidance both reinforce the same operational truth: reliability comes from measuring system behavior continuously, especially during model or prompt changes. You should also personalize responsibly. Never use sensitive categories unless there is a clear legal and product basis; give users control over at least some preference settings; and make it easy to fall back to a neutral, non-personalized mode. ## Delivery model, governance, and productionization roadmap

To make this stable for the long term, you need a platform operating model, not just a codebase. Start by creating versioned contracts for APIs, events, features, and prompts; define service owners; require architecture decision records for durable choices; and set a written deprecation policy so services, fields, or prompt schemas are never removed casually. AWSŌĆÖs operational excellence guidance is directly relevant here: long-term reliability depends on organizing the team, designing the workload, operating it at scale, and evolving it methodically. ŅłĆciteŅłéturn13search0Ņłéturn13search4Ņłéturn13search9Ņłü

For observability, instrument everything with OpenTelemetry from the beginning. At minimum, you want end-to-end traces across request handling, retrieval, ranking, model calls, and downstream storage; metrics for latency, error rate, throughput, and saturation; and structured logs with request, tenant, model, prompt, and feature-flag identifiers. Then put SLOs on the user-critical paths and operate with error budgets, not only uptime anecdotes. That is how you keep an AI-heavy system stable even when underlying models, prompts, or data change. ŅłĆciteŅłéturn11search9Ņłéturn11search19Ņłéturn23search0Ņłéturn23search2Ņłéturn23search7Ņłü

For releases, adopt a multi-gate promotion path: unit tests, contract tests, static analysis, dependency review, secret scanning, prompt and safety evals, offline recommendation evaluation, canary release, and then gradual rollout behind feature flags.
